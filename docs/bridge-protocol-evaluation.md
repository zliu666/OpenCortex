## 一、需求分析

### 核心需求：让 AI Agent 像人一样使用 OpenCortex

| # | 能力 | 描述 | 优先级 |
|---|------|------|--------|
| 1 | 多轮对话 | 保持上下文，连续交互 | P0 |
| 2 | 流式输出 | 实时看到思考和执行过程 | P0 |
| 3 | 工具调用可视化 | 看到 bash/file 等工具执行过程 | P1 |
| 4 | 中断操作 | 发现不对随时停 | P1 |
| 5 | 权限交互 | 安全提示需要外部确认/拒绝 | P1 |
| 6 | 上下文控制 | 子Agent输出不撑爆主上下文 | P0 |
| 7 | 子Agent管理 | 创建/监控/取消子任务 | P2 |
| 8 | 状态查询 | token用量、模型、进度 | P2 |
| 9 | 外部兼容 | 能被其他AI系统调用 | P1 |

## 二、现有协议评估

### 2.1 MCP（Model Context Protocol）

**规范**：Anthropic，2024年11月发布，v2025-03-26
**传输**：STDIO / HTTP+SSE
**核心模型**：工具（Tools）+ 资源（Resources）+ 提示词（Prompts）

| 维度 | 评分 | 说明 |
|------|------|------|
| 多轮对话 | ❌ | 无会话概念，每次调用独立 |
| 流式输出 | ⚠️ | 有 SSE transport，但只用于通知，不用于对话流 |
| 工具可视化 | ❌ | 只返回工具结果，不展示过程 |
| 中断 | ❌ | 无取消机制 |
| 权限交互 | ❌ | 无确认流程 |
| 上下文控制 | ⚠️ | sampling 功能可以部分控制，但不是为上下文压缩设计 |
| 外部兼容 | ✅✅✅ | 行业标准，生态最大 |

**结论**：MCP 适合"暴露 OpenCortex 的工具能力给外部系统"，不适合"Agent 像人一样深度使用 OpenCortex"。

### 2.2 A2A（Agent-to-Agent Protocol）

**规范**：Google，2025年4月发布，v1.0.0
**传输**：HTTP + JSON-RPC 2.0 + SSE
**核心模型**：Agent Card + Task + Message + Part + Artifact

| 维度 | 评分 | 说明 |
|------|------|------|
| 多轮对话 | ✅ | Task 有状态机（submitted→working→completed） |
| 流式输出 | ✅ | 原生 SSE streaming（stream_message） |
| 工具可视化 | ⚠️ | 通过 Part 传递，但无细粒度的工具过程 |
| 中断 | ✅ | 有 cancel_task 操作 |
| 权限交互 | ⚠️ | 支持 human-in-the-loop（task 状态 pending input） |
| 上下文控制 | ⚠️ | Agent 间通过 Message 交换，内容由发送方控制 |
| 外部兼容 | ✅✅ | 开放标准，Google 支持力度大 |
| 子Agent管理 | ⚠️ | 可以委派 Task，但不细粒度 |

**关键设计原则**：
- **Opaque Execution**：Agent 不需要暴露内部思考过程
- **Async First**：原生支持长时间任务
- **Simple**：基于 HTTP + JSON-RPC 2.0

**结论**：A2A 最接近我们的需求。它的 Task 状态机、streaming、cancel、human-in-the-loop 都是我们需要的。

### 2.3 ACP（OpenClaw Agent Communication Protocol）

**规范**：OpenClaw 内部协议
**传输**：STDIO / 进程间通信
**核心模型**：Agent + Session + Task

| 维度 | 评分 | 说明 |
|------|------|------|
| 多轮对话 | ✅ | Session 保持上下文 |
| 流式输出 | ⚠️ | 有推送，但依赖具体 harness 实现 |
| 工具可视化 | ❌ | 不暴露工具执行过程 |
| 中断 | ⚠️ | 可以 kill session |
| 权限交互 | ❌ | 无确认流程 |
| 上下文控制 | ❌ | 全量传递 |
| 外部兼容 | ❌ | OpenClaw 专有 |

**结论**：ACP 适合 OpenClaw 内部调度，不适合深度 Agent 交互。

### 2.4 HTTP API（当前已实现）

| 维度 | 评分 |
|------|------|
| 多轮对话 | ✅ |
| 流式输出 | ❌（待加 WebSocket） |
| 工具可视化 | ❌ |
| 中断 | ❌ |
| 权限交互 | ❌ |
| 上下文控制 | ❌ |
| 外部兼容 | ✅ |

**结论**：够用但粗糙，适合测试。

## 四、方案建议

### 推荐方案：A2A 适配 + 上下文分层

**不造新轮子，基于 A2A 协议扩展**。

理由：
1. A2A 已经覆盖了 80% 的需求（Task 状态机、streaming、cancel、human-in-the-loop）
2. A2A 是开放标准，Google 支持，未来生态会更大
3. 只需要在 A2A 基础上补充两个能力：
   - **工具调用可视化**：通过 A2A 的 Part 扩展
   - **上下文分级摘要**：在 Agent 间传递时控制信息密度

### 架构

```
┌─────────────────────────────────────────────────────┐
│                    调度层（OpenClaw）                  │
│         海星掌柜 → sessions_spawn / ACP               │
└──────────────────────┬──────────────────────────────┘
                       │ ACP（轻量调度）
                       ▼
┌─────────────────────────────────────────────────────┐
│                 OpenCortex A2A Server                 │
│                                                      │
│  ┌─────────┐  ┌──────────┐  ┌──────────────────┐   │
│  │ A2A API │  │ 上下文   │  │ 工具可视化       │   │
│  │ (标准)  │  │ 分层器   │  │ (Part扩展)       │   │
│  └────┬────┘  └────┬─────┘  └────────┬─────────┘   │
│       │            │                  │              │
│       └────────────┴──────────────────┘              │
│                         │                             │
│                    QueryEngine                        │
│                    (核心引擎)                          │
└─────────────────────────────────────────────────────┘

额外暴露 MCP Server（给外部系统用工具）：
┌─────────────────────┐
│  MCP Server (工具层) │  → 暴露 bash/read/write 等工具
└─────────────────────┘
```

### 上下文分层设计（核心创新点）

**问题**：子 Agent 输出全量灌入主 Agent → 上下文膨胀 → 降智
**解法**：三级信息密度

```
Level 0: 原始输出（完整工具输出、思考过程）
         → 存文件/数据库，不进上下文
         → 需要时按需拉取

Level 1: 工作摘要（调了什么工具、结果概要）
         → A2A Message 的 Part 传递
         → 主 Agent 的上下文里只有这个

Level 2: 最终结论（一句话结果）
         → Task 完成时的 Artifact
         → 调度 Agent 只看这个
```

**实现方式**：
- OpenCortex 的 `auto_compact` 模块已有对话压缩能力，复用它
- 工具输出超过阈值（如 50 行）自动生成摘要
- 子 Agent 完成时，只返回 Level 2 给调度 Agent

### 具体实现步骤

#### Phase 1：A2A Server（1-2天）
1. 实现 A2A Agent Card（声明 OpenCortex 的能力）
2. 实现 Task 管理（create/get/cancel/list）
3. 实现 Message 传递（send_message / stream_message）
4. 实现上下文分层器

#### Phase 2：MCP Server（1天）
1. 实现 MCP Tools（bash/read/write/search 等）
2. 注册到 MCP 生态

#### Phase 3：OpenClaw 集成（1天）
1. OpenCortex 注册为 A2A Remote Agent
2. 海星通过 ACP 调度 → OpenCortex A2A Server
3. 端到端测试

#### Phase 4：上下文优化（持续）
1. 摘要质量调优
2. 按需拉取机制
3. 多级缓存

## 五、风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| A2A 规范变动 | 中 | 中 | 关注 Google A2A 仓库，锁定版本 |
| 上下文摘要质量差 | 中 | 高 | 人工评估 + 逐步调优 prompt |
| 性能开销（多层协议转换） | 低 | 中 | A2A 直接 HTTP，不额外转换 |
| OpenClaw 不原生支持 A2A | 中 | 低 | 保留 ACP 调度路径 |

## 七、决策建议

**推荐：方案 A — A2A 适配 + 上下文分层**

优势：
- 不造轮子，站在 Google 的肩膀上
- 覆盖 90%+ 的需求
- 3-4天可完成核心功能
- 未来兼容性好

备选：
- **方案 B**：自建 OCP 协议 — 完全可控，但工作量大（1-2周），且没有生态
- **方案 C**：纯 MCP — 简单但能力不够，只能暴露工具

## 八、附录：A2A 关键概念映射

| A2A 概念 | OpenCortex 对应 |
|---------|----------------|
| Agent Card | OpenCortex 的能力声明（工具列表、模型、安全配置） |
| Task | 一次完整的用户请求处理流程 |
| Message | 用户↔Agent 的一次交互（prompt + response） |
| Part | Message 的组成部分（文本/文件/结构化数据/工具调用） |
| Artifact | Task 的最终产出物 |
| stream_message | 实时推送 Agent 的思考和工具执行过程 |
| cancel_task | 中断当前操作 |
